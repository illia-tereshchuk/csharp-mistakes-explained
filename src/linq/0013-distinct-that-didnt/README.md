# #0013 - Distinct on a Class Without Value Equality

## 💥 Symptom

Subscribers complain about duplicate newsletters. The merge code plainly
calls `.Distinct()` right before sending - the reviewer saw it, approved
it, moved on. And the unit tests are green, because they dedupe lists of
*strings*, and strings dedupe just fine. Only the domain objects don't.

## 🔍 The Offending Code

```csharp
var recipients = fromWebsite.Concat(fromMobileApp).Distinct().ToList();

class Subscriber(string email)
{
    public string Email { get; } = email;   // same email, "different" subscriber
}
```

## 🧠 What's Actually Going On

`Distinct()` doesn't know what "the same" means for your type - it asks
`EqualityComparer<T>.Default`, which uses the type's `Equals` and
`GetHashCode`. A plain class inherits both from `object`, and there
"equal" means **the same reference**. Two `Subscriber` objects with
identical emails are two different pointers, so the dedupe dedupes
nothing.

The sneaky part: the same line of code works flawlessly for `string`,
`int`, records, and anonymous types - all of them define value equality.
This exhibit even proves it against itself: the audit line
`Select(r => r.Email).Distinct()` counts 3, because *strings* compare by
content. One operator, two behaviors, decided entirely by the element
type. And the same contract drives `Contains`, `Except`, `Union`,
`GroupBy`, and every `HashSet` or `Dictionary` key - `Distinct` is just
where it gets noticed.

## ✅ The Fix

Give the type value equality - for a record, that's the whole diff:

```csharp
record Subscriber(string Email);   // Equals + GetHashCode generated by content
```

Full version in [Good.cs](Good.cs) - one keyword changed. The toolbox:

| Option | When it's the right call |
|---|---|
| `record` | Value objects. One keyword, equality and hashing generated |
| `DistinctBy(x => x.Email)` (.NET 6+) | Surgical dedupe by key, no type changes needed |
| `IEquatable<T>` + `GetHashCode` override | Legacy classes you can't turn into records - implement **both** |

## 😈 The Even Worse Sibling

Override `Equals` and forget `GetHashCode`. `Distinct` is built on a hash
set: it buckets elements by hash *first* and only calls `Equals` on hash
collisions. Different default hashes mean your carefully written `Equals`
is **never even called** - the duplicates survive anyway, and now the code
looks fixed. The compiler hints at this with warning CS0659; treat it as
an error.

## 🎓 Advanced Nuance

Anonymous types have value equality built in - which is why the quick
prototype `select new { o.Email }` deduped perfectly, and the bug appeared
only when someone "cleaned up" the code by introducing a proper class. A
refactor that changes no logic, only a type declaration, silently changed
the semantics of every equality-based operator downstream. Records restore
the anonymous-type behavior with a name attached.

## 🔎 How to Find It in Your Codebase

- Every `.Distinct()`, `.Except(`, `.Union(`, `.Contains(` over a class
  type - check the type for equality overrides; if there are none, it
  compares pointers.
- Warning CS0659 ("overrides Equals but not GetHashCode") - promote to
  error.
- Prefer `DistinctBy` with an explicit key: it documents *what* uniqueness
  means at the call site.
